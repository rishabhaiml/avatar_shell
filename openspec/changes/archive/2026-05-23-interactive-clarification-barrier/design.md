## Context

Currently, B.H.A.I. is purely reactive: it wakes up via wake-word, listens, thinks, speaks, and then immediately transitions to `AppState.IDLE`. If B.H.A.I. asks a question or requests clarification, the user must wait for speech playout to finish and say the wake word again. The `interactive-clarification-barrier` introduces a state-aware conversation loop that allows B.H.A.I. to proactive prompt for clarifications and listen for a wake-word-free follow-up response immediately.

## Goals / Non-Goals

**Goals:**
- Implement thread-safe inline parsing of the cognitive persona suffix `[CLARIFY]` in LLM token streaming.
- Extend `config.py` with tracking registers (`WAITING_FOR_CLARIFICATION`, `AppState.FOLLOW_UP_LISTENING`).
- Modify the state-machine loop in `main.py` to route speaking/thinking turns into immediate high-sensitivity listening when the clarification flag is active.
- Safely flush PortAudio audio queues and reset timers at the boundary to eliminate syllable truncation and playout feedback echo leaks.

**Non-Goals:**
- Continuous full-duplex barge-in during regular speaking turns.
- Multi-user conversational tracking.
- Modification of offline transcription models (Whisper/openwakeword weights).

## Decisions

### 1. Suffix-Based Cognitive Persona Tagging
- **Choice**: Use an inline text suffix `[CLARIFY]` generated by the LLM.
- **Rationale**: Keeps the persona prompt lightweight and allows the small 1B model to naturally append the flag when it wants to continue the conversation thread without changing the JSON action format.
- **Alternative**: Multi-model classification of intent (too slow/resource intensive for local CPU runtime).

### 2. State-Aware Follow-Up Listening
- **Choice**: Introduce a separate state `AppState.FOLLOW_UP_LISTENING` in `config.py`.
- **Rationale**: Isolates follow-up logic from standard IDLE/LISTENING states and permits targeted room-decay flushes and immediate clock resets without breaking baseline VAD and wake-word scoring pipelines.

## Risks / Trade-offs

- **[Risk]** Loopback Playout Feedback (Echo triggering fake human input) → **[Mitigation]** The transition to follow-up listening asserts a strict non-blocking hardware flush loop of the `AUDIO_FRAME_QUEUE` and checks physical speaker/TTS status down to absolute zero floor before picking up audio bytes.
- **[Risk]** Clock Truncation Freeze (Max-Cap time limit immediately killing follow-up turns) → **[Mitigation]** Forcefully reset the local epoch timer `listening_start_time` to `time.time()` upon entering the follow-up state.
