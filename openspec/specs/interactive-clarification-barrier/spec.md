## Requirements
### Requirement: Inline Clarification Token Parsing
The system SHALL monitor the token stream during LLM response generation and parse the special `[CLARIFY]` suffix token in the `TokenTextFilter`.

#### Scenario: Suffix Token Is Detected
- **WHEN** the token `[CLARIFY]` is generated by the LLM
- **THEN** the system SHALL set `config.WAITING_FOR_CLARIFICATION` to `True` and strip `[CLARIFY]` from the spoken sentence queue

### Requirement: Proactive State Routing
The system SHALL bypass the default `AppState.IDLE` flow when `config.WAITING_FOR_CLARIFICATION` is True at the end of speaking, and transition directly into the proactive listening state.

#### Scenario: Turn Ends With Clarification Flag Set
- **WHEN** a speaking turn finishes and `config.WAITING_FOR_CLARIFICATION` is True
- **THEN** the system SHALL transition `config.CURRENT_STATE` directly into `AppState.FOLLOW_UP_LISTENING` and re-baseline `listening_start_time` to `time.time()`

### Requirement: Playout Audio Echo Flush
The system SHALL execute a non-blocking hardware audio flush when entering the proactive listening state to prevent loopback echo contamination.

#### Scenario: Proactive Listening Session Starts
- **WHEN** entering `AppState.FOLLOW_UP_LISTENING`
- **THEN** the system SHALL flush the mic `AUDIO_FRAME_QUEUE` and assert a hardware playout floor of zero for both speaker and TTS flags
